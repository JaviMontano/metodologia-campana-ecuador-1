# Anexo F -- Preparacion de Decks y Materiales

**Documento operativo transversal** -- v1.0 (2026-03-29)

> **Aplica a:** todos los servicios del catalogo MetodologIA / JM Labs / IAC.
> **Trigger:** brief de contextualizacion completado (o deadline de preparacion alcanzado sin brief).
> **Responsable:** Facilitador (preparacion de contenido) con validacion del Embajador (logistica y completitud).
> **Regla de autoridad:** este documento define como se preparan los materiales. No modifica alcance ni politicas del canonico. Si hay conflicto, el canonico manda.

## 2) Deck de apertura: estructura y requisitos de contenido

El deck de apertura se usa en la primera sesion de todo servicio. Establece el encuadre, las reglas y el ritmo. Debe ocupar un maximo de 10 minutos de exposicion.

### 2.1 Estructura obligatoria

| Slide | Contenido | Tiempo | Notas para el Facilitador |
|---|---|---|---|
| **1** | Bienvenida + nombre del servicio + logo de la marca | 1 min | Mencionar nombre del Embajador como punto de contacto para logistica. |
| **2** | Reglas del juego | 2 min | Cubrir: (a) datos sensibles prohibidos en IA, (b) grabacion activa y consentimiento, (c) participacion activa esperada (camara encendida, micro para intervenir), (d) canal de soporte durante sesion. |
| **3** | Objetivo de la sesion / programa | 2 min | Usar el objetivo operativo del canonico. Conectar con el reto declarado en el brief. Ser especifico: "Al salir de aqui tendras [entregable concreto]." |
| **4** | Recorrido (modulos / sesiones / fases) | 2 min | Mapa visual del servicio. Para Workshop: 5 modulos. Para Bootcamp: 10 sesiones. Para Programa: 4 fases. Para Consultive: 9 sesiones + playbook final. Indicar donde estamos y hacia donde vamos. |
| **5** | Primer ejercicio (arrancar en <5 min) | 3 min | Instruccion directa: "Abre [Motor IA], crea una conversacion nueva llamada [nombre], y escribe [prompt inicial]." El participante debe estar haciendo algo productivo antes del minuto 10. |

### 2.2 Requisitos de formato

| Aspecto | Especificacion |
|---|---|
| Formato de archivo | PDF para entrega; formato editable (Google Slides / PowerPoint) para el Facilitador |
| Cantidad de texto por slide | Maximo 30 palabras. Priorizar visual + instruccion. |
| Tipografia | Legible a 1080p. Minimo 24pt para cuerpo, 36pt para titulos. |
| Colores | Segun guia de marca de la marca activa (MetodologIA / JM Labs / IAC). |
| Marca | Logo en esquina inferior. Nombre del servicio en header o footer. |
| Idioma | Espanol latinoamericano. Sin anglicismos innecesarios. |

### 2.3 Adaptacion por tipo de servicio

| Tipo | Modificaciones al deck de apertura |
|---|---|
| Workshop | Deck base. Sin modificaciones. |
| Bootcamp | Agregar slide 6: "Como funciona el campus/LMS" (1 min). Agregar slide 7: "Proyecto final -- preview" (1 min). |
| Programa | Agregar slide 6: "Governance y puntos de contacto". Agregar slide 7: "Plan de mentoria". Agregar slide 8: "Capstone -- preview". |
| Consultive | Agregar slide 6: "Como funciona el ciclo de 9 sesiones". Agregar slide 7: "Que vamos a construir juntos (playbook final)". |

## 3) Protocolo de validacion cruzada

El Embajador ejecuta esta validacion **24 horas antes** de la primera sesion (o de cada sesion en servicios multisesion donde sea critico).

### 3.1 Checklist de validacion

| # | Item | Estado | Accion si falla |
|---|---|---|---|
| 1 | Deck de apertura listo y en repositorio | OK / FALTA | Facilitador completa con prioridad maxima. Si no se completa para D-0: usar deck generico de la marca. |
| 2 | Materiales de la sesion listos (plantillas, prompts, asistentes) | OK / FALTA | Facilitador completa. Si faltan plantillas: usar plantillas genericas. Si faltan prompts: el Facilitador los genera en vivo (contingencia aceptable para prompts simples). |
| 3 | Repositorio accesible por participantes | OK / FALLA | Embajador verifica permisos. Si Google Drive falla: compartir materiales por email como backup. |
| 4 | Grabacion configurada en plataforma | OK / FALLA | Embajador configura. Si la plataforma no permite grabar: usar OBS o herramienta local como backup. Notificar a participantes. |
| 5 | Enlace de sesion funcional | OK / FALLA | Embajador genera nuevo enlace y notifica a todos. Tener enlace de backup en otra plataforma. |
| 6 | Checklist 48h respondido por participantes | TODOS / PARCIAL / NINGUNO | Si PARCIAL: Embajador contacta a los faltantes por WhatsApp. Si NINGUNO: Embajador evalua si proceder (minimo: confirmar motor IA y computador). |
| 7 | Motor IA del cliente verificado | OK / BLOQUEADO | Si BLOQUEADO: activar motor alternativo. Si no hay alternativa: preparar caso sin IA + guia post-sesion para replicar. |
| 8 | Intelligence brief compartido con Facilitador | OK / FALTA | Si FALTA: Embajador hace version express (30 min) con datos disponibles. |

### 3.2 Protocolo de contingencia

| Escenario | Accion inmediata | Responsable |
|---|---|---|
| Deck no listo a D-1 | Usar deck generico de la marca con adaptacion minima (nombre del cliente, objetivo) | Facilitador |
| Repositorio inaccesible | Enviar materiales por email a todos los participantes. Resolver acceso post-sesion. | Embajador |
| Grabacion no funciona | Activar grabacion de respaldo (OBS, grabacion local). Priorizar memorias escritas. | Facilitador + Embajador |
| Motor IA bloqueado para todos | Sesion con caso de ejemplo guiado por el Facilitador. Participantes replican post-sesion con su propio motor. Entregar guia de replicacion. | Facilitador |
| Facilitador no disponible (emergencia) | Embajador notifica al cliente. Reagendar con prioridad (primera alternativa en 3 dias habiles). Si hay cofacilitador disponible: ejecutar con cofacilitador. | Embajador + Director |

## 4) Deck de memorias (template de captura durante sesion)

El Facilitador (o un asistente designado) captura decisiones, supuestos y compromisos durante cada sesion. Este template se llena en tiempo real o inmediatamente post-sesion.

### 4.1 Template

```
==========================================================
MEMORIAS DE SESION
==========================================================
Servicio: [Nombre canonico]
Cliente: [Nombre del cliente]
Sesion: [#] de [total] -- [Nombre del modulo/tema]
Fecha: [YYYY-MM-DD]
Facilitador: [Nombre]
Participantes presentes: [Lista o numero]
Participantes ausentes: [Lista o "Ninguno"]
==========================================================

MODULO / BLOQUE 1: [Nombre]
- Decisiones tomadas: [lista]
- Supuestos explicitos: [lista con etiqueta [SUPUESTO]]
- Trade-offs documentados ("que NO haremos"): [lista]
- Compromisos / proxima accion: [responsable + fecha]
- Herramientas/prompts usados: [lista]

MODULO / BLOQUE 2: [Nombre]
- Decisiones tomadas: [lista]
- Supuestos explicitos: [lista]
- Trade-offs documentados: [lista]
- Compromisos / proxima accion: [responsable + fecha]
- Herramientas/prompts usados: [lista]

[Repetir por cada modulo/bloque de la sesion]

==========================================================
RESUMEN DE LA SESION
==========================================================
- Entregables producidos: [lista de artefactos creados]
- Puntos pendientes: [lista]
- Incidentes: [descripcion o "Ninguno"]

==========================================================
PROXIMA SESION (si aplica)
==========================================================
- Fecha: [YYYY-MM-DD]
- Tema: [nombre del modulo/tema]
- Tarea pre-sesion: [que deben traer/hacer los participantes]
- Materiales a preparar: [que debe preparar el Facilitador]

==========================================================
INCIDENTES REPORTADOS (si aplica)
==========================================================
- Incidente: [descripcion]
- Impacto: [como afecto la sesion]
- Resolucion: [que se hizo]
- Escalamiento: [Si/No. Si si: a que nivel, ID de escalamiento]
==========================================================
```

### 4.2 Reglas de captura

| Regla | Detalle |
|---|---|
| Completitud | Cada modulo debe tener al menos 1 decision O 1 compromiso documentado. Si un modulo no produjo ninguno, registrar "Modulo de nivelacion/contexto -- sin decisiones." |
| Inmediatez | Las memorias deben estar completas dentro de las 4 horas posteriores al cierre de la sesion. |
| Formato | Texto plano o Markdown. No requiere formato visual elaborado. La prioridad es la informacion, no la estetica. |
| Almacenamiento | En `03-memorias/sesion-XX/memorias.md` del repositorio del cliente. |
| Revision | El Embajador revisa las memorias antes de compartir con el cliente. Si hay informacion sensible o errores, corrige antes de publicar. |

### 5.2 Convenciones de nombrado

| Elemento | Convencion | Ejemplo |
|---|---|---|
| Carpeta raiz | `{cliente}-{slug-servicio}` en kebab-case | `acme-sa-workshop-de-ocupado-a-productivo` |

| Sesiones | `sesion-XX` con numero de 2 digitos | `sesion-01`, `sesion-02`, ..., `sesion-10` |
| Certificados | `certificado-{nombre-apellido}.pdf` en kebab-case | `certificado-juan-perez.pdf` |
| Versiones de archivos | Sufijo `-vX` si hay multiples versiones | `deck-apertura-v2.pdf` |

### 5.4 Vigencia y cierre

| Hito | Accion | Responsable |
|---|---|---|
| Cierre del servicio | Notificar al cliente la fecha de vencimiento del acceso (incluido en T4) | Embajador |
| Vencimiento (N dias post-cierre, segun canonico) | Revocar acceso del cliente. Archivar carpeta en storage interno. | Embajador |
| Backup | El cliente es responsable de descargar antes del vencimiento. MetodologIA no garantiza recuperacion post-vencimiento. | N/A (responsabilidad del cliente) |
| Archivado interno | Mover a carpeta de archivo interno (no eliminar). Retener por 12 meses adicionales para referencia. | Embajador |

## 6) Checklist de cierre del servicio

> Se almacena en `05-cierre/checklist-cierre.md` del repositorio del cliente. El Embajador lo completa y firma al cerrar la orden.

```
==========================================================
CHECKLIST DE CIERRE
==========================================================
ID de orden: [ODS-YYYY-###]
Servicio: [Nombre canonico]
Cliente: [Nombre]
Fecha de cierre: [YYYY-MM-DD]
Embajador: [Nombre]
Facilitador: [Nombre]
==========================================================

ENTREGABLES
- [ ] Grabacion(es) subida(s) al repositorio.
- [ ] Memorias de todas las sesiones completas.
- [ ] Playbook / entregable principal entregado.
- [ ] Plantillas y prompts entregados.
- [ ] Certificados emitidos y entregados.

COMUNICACIONES
- [ ] T4 (agradecimiento + acceso) enviado.
- [ ] Fecha de vencimiento del repositorio comunicada al cliente.
- [ ] Credito educativo notificado al cliente (monto + vigencia).

CRM
- [ ] Estado de la orden = CERRADA.
- [ ] NPS registrado (o encuesta enviada pendiente de respuesta).
- [ ] Credito educativo registrado (monto + fecha de vencimiento).
- [ ] Siguiente accion registrada (primer follow-up: T5 en D+7).

CALIDAD
- [ ] Incidentes documentados (o "Sin incidentes").
- [ ] Feedback del Facilitador capturado.
- [ ] Lecciones aprendidas registradas (si aplica).

FIRMA
Embajador: _______________________ Fecha: ___________
==========================================================
```
